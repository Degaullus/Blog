Micropayement and Websites

    techNews, 2024.02.14

Until now, there are basically two famous ways to make money with your website: sell some space for ads (and get money from them) or create your paywall and manage subscriptions.

According to The Register, Chrome engine developers are working on micropayment prototypes. In a few words, any request will generate a small amount of cash (very small, like cents). The principle is easy: Small amounts and no active user interaction for it. Just surfing.

How could it work? Simply by endowing users and creators with a digital wallet and using an open Payment API. Hehe, we know this one right ? Every click/request enables a small microtransaction. Your .get and ${} are really starting to make sense.

The concept is not new; some serious discussions already started in the '90s: In the past, some voices raised and shouted that micropayment will be the future. But, you know, skepticism won. Today, the development of crypto, blockchaining, and decentralized wallets seems to give a new start to this thing.

Some notions about API

We just started with API. before we get lost, let's talk about certain notions bro.

THE ENDPOINT is the moment where you can write an interaction with an API. Like sumoning it in dungeons and dragons. Love it. You have some option for this : "get", "put", "post" and "delete"(request method). This is the end of the URL, full of power,

Basicelly, they are powerfull sumoning tools. Some of them less. For example, with the request "get", you will be able to get the information (like the users, mails, ect). But with some other, who sounds more like dark magic spells, you can create inside the website (like with the request "post"). The

So biggest questions : What is an API (comuunication bet etwen client app and server app), what is an Endpoint () ? What is a request ?

Client: This is the application or system that initiates the request. It could be a web browser, a mobile app, a server, or any other software component that needs to interact with the API. Server: This is the system hosting the API. It receives requests from clients, processes them, and sends back responses containing the requested data or performing the requested actions. HTTP Protocol: API requests are typically made over the Hypertext Transfer Protocol (HTTP) or its secure version, HTTPS. HTTP defines a set of methods (such as GET, POST, PUT, DELETE) that indicate the desired action to be performed on a resource located on the server. Endpoint: An endpoint is a specific URL that represents a resource on the server. API requests are sent to specific endpoints to perform actions or retrieve data related to that resource. For example, /users might be an endpoint to retrieve a list of users, and /products/123 might be an endpoint to retrieve information about a specific product with ID 123. Request Method: The request method specifies the type of action the client wants to perform on the resource. The most common methods are: GET: Retrieve data from the server. POST: Send data to the server to create a new resource. PUT: Update an existing resource on the server. DELETE: Remove a resource from the server. Request Headers: Headers contain additional information about the request, such as authentication credentials, content type, or cache control directives. Request Body: For some request methods, such as POST and PUT, the client may need to send additional data in the request body. This data could be in various formats, such as JSON, XML, or form data, depending on the API's requirements. Response: After receiving and processing the request, the server sends back a response to the client. The response typically includes a status code indicating the outcome of the request (e.g., 200 for success, 404 for not found, 500 for server error) and, optionally, a response body containing the requested data or an error message.
